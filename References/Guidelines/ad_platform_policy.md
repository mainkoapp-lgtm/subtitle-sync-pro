# 📢 데스크톱(EXE) 및 웹 서비스 광고 운영 전략 (2026.04)

본 문서는 SubFast 서비스의 수익 창출을 위해 데스크톱 추출기(EXE)와 웹 서비스에 각각 최적화된 광고 플랫폼을 적용하는 운영 전략을 정의합니다.

## 0. 최종 운영 결정 (2026-04-20)
*   **핵심 원칙**: **[수익 절대 보전]** 광고 형태(영상/이미지)와 관계없이, **광고 수익이 AI API 비용을 반드시 상회**해야 함.
*   **게이트웨이 로직**: 사용자가 [싱크 맞추기] 버튼을 누르면 **광고 노출이 무조건 선행**되어야 하며, 광고 수익 발생이 보장된 시점에만 AI API 로직이 실행됨. (수익 없는 API 소모 금지)
*   **추출기 (.exe)**: **클릭몬 (ClickMon)** 또는 **Monetag (Direct Link)** 탑재
*   **웹 서비스 (Browser)**: **Monetag (Interstitial)** 및 **쿠팡 파트너스** 병행
*   **API 수익성**: 최저 단가 광고 노출만으로도 **API 비용 대비 최소 3배~100배 수익** 확보 가능 확인
*   **AI API 엔진**: **Google AI Studio (Gemini 3.1 Flash-Lite)** - 배포자 키 사용 (광고 시청 후 해제)

## 2. 상세 분석 및 근거

### A. 클릭몬 (ClickMon) - [공식 확인 성공]
*   **특징**: 국내 환경에서 EXE 프로그램에 광고를 넣을 때 가장 현실적이고 안정적인 플랫폼입니다.
*   **공식 정책**: 실시간 광고 노출 기록과 함께 '응용프로그램' 카테고리를 제공합니다. 프로그램 설치 경로와 실제 구동 화면(스크린샷)을 통해 심사를 진행합니다.
*   **장점**: 웹뷰(WebView)뿐만 아니라 간단한 HTTP 요청으로 광고 데이터를 받아올 수 있어 프로그램이 가볍습니다.

### B. 구글 애드센스 (Google AdSense) - [주의 사항]
*   **금지 규정**: [AdSense 도움말 - 소프트웨어 애플리케이션의 광고](https://support.google.com/adsense/answer/4450771)에 따르면 "광고는 웹페이지가 아닌 소프트웨어 애플리케이션 내에서 표시될 수 없다"고 명시되어 있습니다.
*   **우회 위험**: WebView를 사용하여 웹사이트를 띄우고 그 안에 애드센스를 넣는 방식도 '프레임 내 노출' 또는 '소프트웨어 내 노출'로 간주되어 정책 위반 대상이 됩니다.

### C. 모비온 (Mobon)
*   **특징**: 광고주 도달 범위가 넓고 PC용 소프트웨어(유틸리티, 배너) 매체사에 대한 보상 체계가 잘 갖춰져 있습니다.
*   **단점**: 일정 수준 이상의 사용자(트래픽)가 보장되어야 매체 승인이 유리합니다.

## 3. 권장 구현 아키텍처 (Bridge Page Strategy)

정책 위반 리스크를 최소화하고 광고 교체를 용이하게 하기 위해 아래 구조를 권장합니다.

1.  **브릿지 페이지 구축**: 본인의 서버(예: `subfast.com/ads/banner_v1`)에 광고 전용 HTML 페이지를 생성합니다.
2.  **광고 코드 삽입**: 해당 HTML 페이지 내에 클릭몬이나 모비온의 광고 스크립트를 삽입합니다.
3.  **추출기 앱(EXE)**: 앱 내부의 특정 영역에 **WebView2** 또는 **Iframe**을 배치하고, 위 브릿지 페이지 URL을 로드합니다.
4.  **쿠팡 파트너스 병행**: 클릭몬 광고 하단에 쿠팡 파트너스 배너를 함께 배치하여 수익을 극대화할 수 있습니다. (앱 직접 노출이 아닌 웹 노출 형식이 되어 안전함)

### 6. 구현 완료된 보안 아키텍처 (Step 1~3)
#### [토큰 기반 API 접근 정책 (Token-First Policy)]
- **정의**: '토큰'은 단순한 기능이 아니라 서비스 전체의 **접근 권한** 자체로 정의함.
- **구조**:
    1. 광고 시청 완료 (SSV 검증 예정)
    2. 서버에서 32바이트 보안 토큰(`Job Token`) 발행 (유효기간 5분)
    3. 사용자가 토큰을 포함하여 싱크 API 요청
    4. 서버에서 토큰 유효성 검증 및 즉시 소모(One-time 사용)
    5. 검증 통과 시에만 서버 마스터 API 키 주입 및 로직 실행

#### [UI/UX 최적화]
- **배포 모드 UI**: 일반 사용자에겐 불필요한 AI 엔진 정보 및 API 키 입력란을 완전히 숨김(Hidden).
- **광고 브릿지**: 싱크 시작 전 5초간의 안내 모달을 통해 광고 시청을 유도하고, 작업 티켓 발행 프로세스를 동기화함.

#### [보안 성과]
- **Key Hiding**: 클라이언트 소류 코드에서 Gemini API 키 완전 제거.
- **Anti-Forgery**: 클라이언트의 가짜 신호가 아닌 서버가 발행한 일회성 토큰이 있어야만 API 호출 가능.

3.  **수익성 및 꼼수(Abuse) 방지**:
    *   **실시간 수익 확정 불가 원칙**: 기술적 한계(정산 주기)를 인정하되, **'보상 지급 완료 이벤트 = 수익 인정'**으로 간주하여 실행 권한 부여.
    *   **새로고침/이탈 대응**: 토큰이 DB/Redis에 저장되므로, 새로고침 시에도 유효시간(5분) 내라면 작업 재개 가능 (단, 작업 시작 시 소모됨).

    ```
    광고 시청 → [완료] → 플랫폼 서버 신호(SSV) → 백엔드 검증 → Job Token 발급 → 싱크 API 호출(Token 포함) → 작업 시작 및 Token 폐기
    ```
